# Kint 統合仕様書

## 1. 文書目的
本書は、Kint の業務仕様・システム仕様・API仕様・データ仕様を統合し、
実装担当、レビュー担当、運用担当が同じ前提で作業できるようにするための基準文書である。

## 2. 対象範囲
- 対象システム: NFC 勤怠管理システム Kint
- 対象機能:
  - Web アプリでの打刻（WebUSB + PaSoRi）
  - カード忘れ時のユーザー検索フォールバック打刻
  - 勤怠一覧・勤怠修正・変更履歴表示
  - iCal シフト連携
- 対象外:
  - 旧 Desktop 打刻アプリの新規機能追加

## 3. システム構成
- Server: FastAPI + SQLite
- Client: React SPA (打刻 UI + 管理 UI)
- NFC: WebUSB を通じて PaSoRi (RC-S380/RC-S300) を利用

参照:
- アーキテクチャ: docs/architecture.md
- API 契約: docs/api-contract.openapi.yaml
- DB 設計: docs/database-design.md
- 実装計画: docs/implementation-plan.md

## 4. 利用者と権限
- 従業員:
  - 自分の打刻実行
  - 自分の勤怠修正
  - 自分の勤怠変更履歴閲覧
  - マイページで自分のプロフィール編集（メールアドレス、表示名、氏名）
- 管理者:
  - ユーザーの登録・修正・削除
  - 全従業員の勤怠参照・修正
  - カード登録
  - シフト同期実行
  - 全従業員の変更履歴閲覧
  - システム設定の参照・変更

### 4-1. 初期登録アカウントと保護
- **manager (ID: manager)**:
  - システムの初期管理者（admin）ユーザー。初回起動時に自動作成される。
  - 別の管理者アカウントを設定した後は、セキュリティのために無効化（非アクティブ化）することが推奨される。
- **system (ID: system)**:
  - 自動退勤等のスケジュール処理実行や監査ログの記録アクターとして使用されるシステム内部専用ユーザー。
  - システム動作上必須のアカウントであるため、ユーザー管理画面のUI上で「削除」および「完全削除」の操作ボタンが非表示となるよう制限されている。

## 5. 機能仕様

### 5-1. 打刻機能
- 打刻ページへのアクセス:
  - 打刻ページはログインなしで利用できる（認証不要）。
  - Web アプリのトップページには「打刻」「ログイン」の 2 ボタンを表示し、打刻ページへはログインなしで直接遷移できる。
- 通常打刻:
  - WebUSB で PaSoRi を接続し、カードの IDm を取得して打刻する。
  - 打刻完了時、表示時間の経過を待たずに即座に次のカード読み取りを開始（ポーリングを再開）し、新しい打刻が検出されたら即座に既存の表示メッセージを消去して新たな打刻処理を開始する（割り込み打刻）。
  - ただし、打刻完了または打刻取り消しの直後に、クールダウン期間内で同一ユーザーが再度打刻しようとした場合は、直近打刻（物理削除された取り消し打刻の時刻もメモリキャッシュから解決）を基に連続打刻（二重打刻）を制限する。
  - 制限の内容として、直近の打刻から10秒以内の連続打刻については、エラーを出さずに無視（正常系レスポンスを返すが、DBの更新は行わない）する。
  - 10秒を超え、設定されたクールダウン時間以内の打刻の場合は、連続打刻（二重打刻）エラーを検出・防止し、直前の打刻結果（成功メッセージ）は消去せずに画面に残したまま、エラーメッセージを追加表示する。このエラーメッセージは、バックエンドから返された残りのクールダウン秒数の経過後に自動的に消去される。
- フォールバック打刻:
  - カード忘れ・読取不能時は表示名/氏名の一部でユーザー候補を検索し、選択した user_id + reason で打刻する。
- 打刻判定:
  - 当日レコード未作成なら check_in。
  - 当日 check_in のみ存在なら check_out。
  - 退勤打刻時に「シフト終了時刻 + シフト超過許容時間」を過ぎている場合、打刻画面にて「許可済みの超過勤務」としての申請（理由入力）を求めます。申請を行わずに打刻を完了させることも可能で、その場合はシフト終了時刻への丸めが適用されます。
  - 不正遷移は 409 を返す。
- 打刻の自動取り消し:
  - 出勤打刻の後、5分（300秒）以内に退勤打刻を行った場合、その出退勤記録（および紐づく監査ログ）を自動的にデータベースから削除（取り消し）する。
  - システム設定の連続打刻クールダウン制限（`punch_cooldown_seconds`）は適用されたままとする（クールダウン期間が経過した後に取り消しが可能）。
  - 打刻画面には、打刻を取り消した旨を青色のアラート表示で示す。
- 効果音の再生:
  - 打刻成功および打刻エラーの表示時、ブラウザ側で効果音（ビープ音）を再生する。
  - 効果音は外部の音声アセットに依存せず、HTML5標準の Web Audio API を使用してクライアント側で動的に合成する。
  - 再生パターンは以下の通りとする：
    - 出勤成功 (`check_in`): 「ピピッ」と高い電子音を2回再生（880Hz, 80msの間隔）
    - 退勤成功 (`check_out`): 「ピプー」と高い電子音から中音へ下がる音を再生（880Hz, 80ms -> 587.33Hz, 200ms）
    - 打刻取消 (`cancelled`): 「ピッ」と短い電子音を1回再生（880Hz, 150ms）
    - 打刻エラー: 「ブー」と低い警告音（のこぎり波）を1回再生（180Hz, 350ms）
  - ブラウザの自動再生ポリシーに対処するため、「PaSoRi に接続」ボタンのクリック、および「カード忘れ打刻」の打刻ボタン送信のタイミングで `AudioContext` をあらかじめ初期化・有効化しておく。


### 5-2. カード登録機能
- Web アプリから card_idm を user_id に紐付ける。
- 同一 card_idm の複数ユーザー共有は禁止。
- 1ユーザー複数カードは許可。

### 5-3. 打刻修正申請機能
- 修正時は reason 必須。
- 本人は自分の打刻のみ修正申請可能（打刻修正申請）。
  - 一般従業員の「打刻修正申請」は、1分単位で時刻を指定でき、理由は必須（空欄や空白スペースのみの入力はエラーとして拒否する）となる。
- 管理者は全勤怠を直接修正可能。
  - 直接修正時の時刻入力は5分刻みとし、理由は任意。
- 修正申請時、出勤希望時刻から5分以内に退勤希望時刻を設定するような申請（勤務時間が5分以下になる修正）は受け付けない（エラーを返してブロックする）。

### 5-4. 変更履歴機能
- 勤怠修正時は必ず変更履歴を追記保存する。
- 履歴は不変ログとして扱い、更新・削除しない。
- 履歴には before/after、reason、actor、changed_at を含む。

### 5-5. シフト連携
- iCal URL からシフトを取得し DB に保存。
- 定時自動同期: `shift_sync_time` 設定に指定した時刻に、APScheduler が毎日 1 回自動で同期を実行する。
- 手動同期:
  - バックグラウンド非同期実行: `POST /api/v1/shifts/sync` で開始可能（非同期受理 202）。
  - 即時ブロッキング実行: `POST /api/v1/shifts/sync/now` を呼び出すことで同期を実行し、完了後の統計結果（件数など）やエラー詳細を即座に取得可能（成功時 200 / 失敗時 400）。
- `shift_sync_time` が未設定（null / 空文字）の場合は自動同期を実行しない。

### 5-6. ユーザー管理機能（管理者専用）
- ユーザー登録:
  - 管理者はユーザーを新規作成できる。
- ユーザー修正:
  - 管理者はユーザーの氏名、表示名、メールアドレス、ロール、有効状態を更新できる。
- ユーザー削除:
  - 通常は監査整合性のため論理削除（`is_active=false`）とする。既存の勤怠・履歴データは保持する。
  - 管理者が「完全削除（物理削除）」を実行した場合は、そのユーザーに関連するすべてのデータ（出退勤レコード、修正申請、カード、シフト情報、変更ログなど）をDBから物理的に削除する。
  - 削除対象ユーザーがシステム設定（`system_settings`）の最終更新者（`updated_by_user_id`）である場合は、データベースの外部キー制約エラーを回避するため、自動的に更新者をシステム用の `system` ユーザーに書き換えた上で物理削除を行う。
- NFCカードの管理・登録:
  - 管理者は、ユーザー管理画面から任意の従業員のカード情報一覧を取得・参照できる。
  - 管理者は、ユーザーのカード名変更およびカード削除を管理画面上で行うことができる。
  - 管理者は、その場でPaSoRi（WebUSB）を使用してカード（IDm）を読み取り、任意の従業員に紐付けて新規登録することができる。
- ユーザー情報の一括保存・復元（バックアップ・リストア）:
  - 登録されているすべてのユーザー情報および紐づくNFCカード情報をJSON形式でローカルファイルに一括保存（エクスポート）できます。
  - JSONファイルからユーザーおよびカード情報を一括復元（インポート）できます（UPSERT処理）。
  - インポート時にエラー（メールアドレスやカードIDmの重複、入力データ不正など）が発生したユーザーは、そのユーザーの変更のみをロールバックしてスキップし、他の正常なユーザーの処理を継続します（部分インポート）。
  - インポートに含まれない既存のこのユーザーのカードは物理削除されます。インポートに含まれない既存ユーザー自体は削除も無効化もされずそのまま維持されます。

### 5-8. システム設定機能（管理者専用）
- 管理者はシステム設定画面から以下の設定値を変更できる。
  - 連続打刻クールダウン秒数（`punch_cooldown_seconds`）: 0〜3600 秒
  - シフト開始前打刻の許容時間（`shift_checkin_early_minutes`）: 0〜120 分
  - シフト超過許容時間（`overtime_allowance_minutes`）: 0〜120 分
  - シフト iCal 同期 URL（`shift_ical_url`）
  - iCal 自動同期時刻（`shift_sync_time`）: 「時」と「分」のセレクトボックス。バックエンドへの保存値は HH:MM 形式。未選択で自動同期 OFF。
  - シフトカレンダー即時同期: 設定画面上に「直ちに同期する」ボタンを提供し、クリック時に即座に iCal データの同期を実行。成功時は追加・更新・削除・スキップの件数を、失敗時はエラーメッセージ（取得失敗や設定不足等）をその場に表示する。
  - サイト名（`site_name`）: ヘッダーやログイン画面等に使用されるウェブアプリ全体の名称。1〜50文字。
  - サイトのサブタイトル（`site_subtitle`）: トップページ等に表示されるサブタイトル。1〜100文字。
  - 打刻結果表示時間（`punch_result_display_seconds`）: 打刻成功メッセージおよびエラーメッセージを画面に表示しておく秒数。1〜300秒（デフォルト: 30秒）。
  - 月次レポート自動通知時刻（`monthly_report_time`）: 「時」と「分」のセレクトボックス。トグルONのとき月末日のこの時刻に従業員へ月次勤務実績を自動メール通知します。トグルOFFで自動通知無効。
  - ログイン継続時間（`login_token_expire_hours`）: ログインセッション（JWTアクセストークン）の有効期限。1〜8760時間（デフォルト: 168時間 [7日間]）。
  - Google新規登録の許可（`enable_google_signup`）: Googleログイン時の新規ユーザー自動登録（アカウント作成）のON/OFF。デフォルトは ON (許可)。無効（OFF）にすると、登録済みのユーザーのみGoogleログインが可能になり、未登録のGoogleアカウントからの新規登録は拒否されます。

- 設定値は DB に永続化し、サーバー再起動後も保持される。
- DB に値がない場合は環境変数のデフォルト値にフォールバックする。
- 変更は即時反映される（サーバー再起動不要）。
- `shift_sync_time` 変更時はスケジューラーのジョブが即時リスケジュールされる。
- **データベースのフルバックアップ・復元**:
  - 管理者は現在のデータベース（SQLite）全体のバックアップファイルをダウンロードできます。
  - 管理者はダウンロードしたバックアップファイルをアップロードすることで、データをオンラインで上書き復元できます。復元前にはデータが完全に上書きされる旨の注意喚起と確認ダイアログが表示されます。
  - アップロードされたファイルは整合性チェック（`PRAGMA integrity_check`）および必要なテーブル構造（`users`, `attendances`, `cards`）の存在チェックが行われ、不正なファイルは拒否されます。


### 5-7. マイページ機能（本人専用）
- ログイン済みユーザーはマイページで以下を自分で編集できる。
  - メールアドレス
  - 表示名
  - 氏名
- 権限境界:
  - マイページから role / is_active / 他ユーザー情報は変更できない。
  - 管理者であってもマイページ操作は「本人情報の更新」に限定する。
- 監査要件:
  - メールアドレス、表示名、氏名の変更は監査ログを保存する。
- UX 要件:
  - 更新成功時は即時に最新プロフィールを再表示する。
  - メールアドレス変更時は再ログインを要求する（セッション無効化）。

### 5-9. 管理者用勤怠一覧・CSV出力・月次サマライズ機能
- **管理者用勤怠一覧**:
  - 管理者は、全従業員の指定月の勤怠レコードおよびシフト（Google カレンダー同期データ）を突き合わせた月次日別一覧を閲覧可能。
  - これにより、シフトとの乖離（遅刻、早退、欠勤、未打刻など）がひと目で確認できる。
    - 未来のシフト予定日（日本時間における本日より未来の日付で、シフトが登録されており、かつ打刻がない日）については、ステータス（状態）は「欠勤」ではなく「出勤予定」として表示されます（このとき「欠勤日数」の月次集計にはカウントされません）。
- **1ヶ月単位でのサマライズ（月次集計）**:
  - 従業員別に当月の「所定日数」「出勤日数」「欠勤日数」「不整合件数」「総勤務時間」「4月からの総勤務時間」を集計する（勤務時間は画面表示上、`[h]:mm` 形式で表示される）。
  - 集計ロジックはシフト時間およびシステム設定に基づき、サーバーサイド（Service層）で一貫して算出され、一般従業員はマイページ上等で自己のサマリーのみ閲覧を許可する。
  - **従業員検索・絞り込み**: 管理者は、表示名、氏名、またはメールアドレスを入力することで、月次サマリー一覧の従業員をリアルタイムに絞り込み検索できる。
- **ローカルファイル（CSV出力）保存**:
  - 管理者は指定年月・全従業員の「勤怠詳細（日別データ）」または「月次サマリー（従業員別集計データ）」を UTF-8 (BOM付き) の CSV 形式でダウンロードし、ローカル環境（Excel 等）で保存・二次利用ができる。CSV内の勤務時間は、Excelで `[h]:mm` 形式として解釈できる `h:mm` 形式で出力され、ヘッダーから `(h)` 単位表記は除去される。

### 5-12. 月次勤怠レポートの自動メール通知機能
- **自動メール通知の概要**:
  - 管理画面で設定された `monthly_report_time`（デフォルト値 `20:00`）に従い、毎月末日の指定時刻に自動的にバックエンドで集計および従業員宛てのメール通知を実行します。
  - 対象者は、メールアドレスが設定されているアクティブな従業員とし、管理者（`role == 'admin'`）は除外されます。
- **通知するデータ項目**:
  - **1か月ごとの勤務日数**: 当月中に打刻（出勤・退勤）が行われた総日数。
  - **1か月ごとの勤務時間**: 当月の勤務時間（5分単位丸め適用後）の合計（メール本文中では `[h]:mm` 形式で表示）。
  - **4月からの総勤務時間**: 当年4月1日（または前年4月1日）から当月末日（実行日）までの全期間における勤務時間の総計（メール本文中では `[h]:mm` 形式で表示）。
- **動作と制御**:
  - 管理者は「システム設定」画面から自動通知のON/OFFおよび自動通知時刻（`monthly_report_time`）を「時」と「分」のセレクトボックスで編集・保存できます。
  - 通知時刻が保存されると、即座に APScheduler の定期実行スケジュールが月末日の指定時刻で更新されます。トグルがOFF、または未設定の場合は自動通知が無効化されます。

### 5-10. 個別ユーザー宛てお知らせ機能
- **自動作成条件**:
  - 申請された勤怠修正申請が却下された際、申請した従業員に対して却下コメントを含めたお知らせ（「勤怠修正申請が却下されました。理由: ○○」）を自動的に作成・通知する。
- **表示と操作（ログイン時）**:
  - ログイン時、画面上部のナビゲーションバーにお知らせベルアイコンを表示し、未読件数を赤いバッジで提示する。
  - ベルアイコンをクリックすることで、Glassmorphism調のプレミアムデザインを施したポップオーバーにてお知らせ一覧（作成時間からの相対時間表示付き）を表示する。
  - 各お知らせ項目に対して個別既読化でき、全体に一括既読化するボタンを設置する。
- **表示と操作（未ログイン打刻時）**:
  - 従業員がカード打刻等を行う際、該当ユーザー宛てに未読のお知らせが存在する場合、打刻結果画面においてお知らせの存在を促す警告アラート（ログインして確認することを推奨するメッセージ）を表示する。
- **データクリーンアップ（自動削除）**:
  - 肥大化防止のため、作成から180日を経過したお知らせレコードは、毎日 04:05 にバックエンドで自動的にクリーンアップ（削除）される。
- **カスケード削除**:
  - ユーザーの登録が完全削除（データベース物理削除）された場合、そのユーザーに紐付いているすべてのお知らせは自動的に連動削除（カスケード削除）される。

### 5-11. 管理者による勤怠追加・編集・削除機能
- **勤怠記録の手動追加**:
  - 管理者は、締め処理（ロック）が行われていない期間に限り、特定の従業員に対して任意の日の勤怠レコードを直接追加できる。
  - シフトや打刻がない日も含めて追加でき、すでに勤怠レコードが存在する日であっても追加で別の時間帯のレコードを手動登録可能（複数レコードの追加をサポート）。
  - 手動追加時の打刻元（`source`）は `'admin_manual'` として記録され、追加理由（`reason`）の入力が必須となる。また、初回作成の監査ログが、変更前（`before`）を None として `AttendanceChangeLog` に同一トランザクションで記録される。
  - 時間重複チェックが働き、重複があった場合は日本標準時（JST ローカル時刻）に変換した時刻表記でエラー警告を表示する。
- **勤怠記録の手動編集（直接修正）**:
  - 管理者は、締め処理（ロック）が行われていない期間に限り、特定の勤怠レコードの勤務時間を直接修正（PATCH）できる。
  - 直接修正すると、そのレコードは手動編集扱い (`is_manual_work_time = True`) となり、打刻時間からの自動丸め計算（つくまなルール）は行われず、管理者が直接設定した勤務出勤/退勤時間（`work_start`/`work_end`）が最優先で保持される。
  - 手動編集時の時刻入力は5分刻みのドロップダウンから選択し、修正理由は任意（オプショナル）となる。
  - 自動計算に戻すオプション（`reset_to_auto`）を選択して保存することで、手動フラグが解除され (`is_manual_work_time = False`)、実際の打刻時間から再計算されるようになる。
- **勤怠記録の削除**:
  - 管理者は、締め処理（ロック）が行われていない期間に限り、特定の勤怠レコードを削除できる。
  - **打刻データが存在するレコードの削除**:
    - その日に打刻データが存在する場合、勤怠レコード自体の物理削除は行わない。代わりに、`work_start` と `work_end` を `NULL` にクリアし、`is_manual_work_time = True`（勤務時間 0h）とすることで、実際の打刻記録を残したまま勤務時間のみをクリア（ソフトデリート）する。
  - **打刻データが存在しないレコードの削除**:
    - シフト予定しかなく打刻データが存在しない日に手動追加された勤務時間を削除する場合などは、勤怠レコード自体をデータベースから物理的に削除する。
  - 物理削除された場合は、そのレコードに紐付く変更履歴（`AttendanceChangeLog`）および申請情報も同時に物理削除（クリーンアップ）される。

### 6-2. 打刻リクエスト
- oneOf 条件:
  - card_idm + device_id + occurred_at
  - user_id + reason + device_id + occurred_at
- カード忘れ打刻の候補検索:
  - GET /api/v1/punches/users?q=<表示名/氏名/ユーザーIDの一部>
  - 返却対象はアクティブユーザーのみとし、候補から選択した user_id を打刻に利用する。

### 6-3. エラー仕様
- 401: 認証失敗
- 403: 権限不足
- 404: カード未登録 / ユーザー未登録 / 対象なし
- 409: 二重打刻または不正遷移
- 422: リクエストバリデーションエラー
- 共通レスポンス形式: code / message / detail

### 6-4. ユーザー管理 API バリデーション仕様
- 共通ルール:
  - アカウントID（`id`）はメールアドレス（`email`）と同じ値である必要があり、文字数は 3〜254 文字とします。
  - email は RFC 準拠フォーマットかつ一意であること。
  - name は 1〜50 文字、full_name は 1〜100 文字。
  - role は admin / employee のみ。
  - 文字列項目は前後空白を除去して評価する。
- POST /api/v1/users:
  - 必須: id, name, full_name, email, role。※ `id` と `email` が一致していない場合は 422 バリデーションエラーを返します。
  - 既存メール（およびアカウントID）重複は 409 を返す。
  - 入力違反は 422 を返す。
- PATCH /api/v1/users/{user_id}:
  - 更新対象フィールドが 1 つ以上必要（空 body は 422）。
  - email 更新時は重複チェックを行い、重複は 409。
  - role 更新時は admin / employee 以外を拒否（422）。
  - is_active=false へ更新する場合、最後の有効な admin を無効化してはならない（409）。
- DELETE /api/v1/users/{user_id}:
  - 通常は論理削除（is_active=false）として扱う。
  - クエリパラメータとして `hard=true` が指定された場合は物理削除（完全削除）を行う。
    - 物理削除時は、そのユーザーに紐づくすべてのデータ（カード、勤怠、ログ、申請など）をクリーンアップする。
    - システム設定の更新者（`updated_by_user_id`）に削除対象ユーザーが設定されている場合は、自動的に `system` ユーザーへ付け替えた上で削除を実行する。
  - 既に is_active=false のユーザーに対する論理削除は冪等に 204 を返す。
  - 最後の有効な admin の削除要求は 409 を返す。
  - `system` ユーザーはシステム動作に不可欠なため、UIからの削除操作を制限（非表示化）している。

### 6-5. マイページ API バリデーション仕様
- GET /api/v1/me:
  - 認証済みユーザー自身のプロフィールを返す。
- PATCH /api/v1/me/profile:
  - 更新可能: email, name（表示名）, full_name（氏名）。
  - 更新対象フィールドが 1 つ以上必要（空 body は 422）。
  - email は RFC 準拠フォーマットかつ一意であること（重複は 409）。
  - name は 1〜50 文字、full_name は 1〜100 文字。
  - 文字列項目は前後空白を除去して評価する。
  - email を変更した場合は現在セッションを無効化し、401 で再認証を要求する。

### 6-6. システム設定 API バリデーション仕様
- GET /api/v1/settings:
  - admin ロールのみアクセス可能。
  - DB 未設定のキーは環境変数またはデフォルト値を返す。
- PATCH /api/v1/settings:
  - admin ロールのみアクセス可能。
  - 更新対象フィールドが 1 つ以上必要（空 body は 422）。
  - `punch_cooldown_seconds` は 0 以上 3600 以下の整数（422 で拒否）。
  - `shift_checkin_early_minutes` は 0 以上 120 以下の整数（422 で拒否）。
  - `shift_ical_url` は文字列 (URI 形式) または null / 空文字列（未設定として扱う）。
  - `shift_sync_time` は `HH:MM` 形式（24時間表記）または null / 空文字列（自動同期 OFF）。形式不正時は 422。
  - `site_name` は 1〜50文字の文字列（形式不正または範囲外は 422）。
  - `site_subtitle` は 1〜100文字の文字列（形式不正または範囲外は 422）。
  - `punch_result_display_seconds` は 1 以上 300 以下の整数（422 で拒否）。
  - `login_token_expire_hours` は 1 以上 8760 以下の整数（422 で拒否）。
  - `enable_google_signup` は真偽値 (bool)（形式不正時は 422）。

詳細は docs/api-contract.openapi.yaml を正とする。

## 9. 障害時仕様
- WebUSB 非対応環境検出時:

  - 再接続ガイドを表示
  - 代替打刻（ユーザー候補選択 + reason）を許可

## 10. 受け入れ条件（システム）
- Web アプリのトップページに「打刻」「ログイン」ボタンが表示される。
- ログインなしで打刻ページを開き、NFC 打刻および代替打刻（ユーザー候補選択 + reason）ができる。
- 管理者がユーザーを登録/修正/削除（論理削除）できる。
- ログイン済みユーザーがマイページで自分のメールアドレス、表示名、氏名を更新できる。
- マイページ経由では role / is_active / 他ユーザー情報を更新できない。
- メールアドレス変更後に再ログインが要求される。
- WebUSB で IDm を取得して打刻できる。
- 表示名/氏名の一部で検索した候補を選択し、reason とともに代替打刻できる。
- source が webusb_nfc / web_user_id で正しく保存される。
- 勤怠修正時に履歴が欠落なく追加される.
- 管理画面で source と履歴が確認できる。
- 管理者は全従業員の指定月の勤怠サマリー（出勤日数、欠勤日数、不整合件数、総勤務時間、4月からの総勤務時間）を一覧表示・検索（表示名、氏名、メールアドレス）できる。
- 管理者は指定月の勤怠データ（ユーザー別日別詳細、ユーザー別月次サマリー）を CSV ファイル（BOM付きUTF-8、勤務時間は `[h]:mm` 形式）としてローカル保存できる。
- 従業員本人はマイページから自己の月次サマリー情報を取得・参照できる。
- 管理者が登録ユーザー情報とカード情報を一括で JSON ファイルに保存（エクスポート）できる。
- 管理者が JSON ファイルから登録ユーザー情報とカード情報を一括復元（インポート）できる。その際、エラー発生したユーザーのみスキップされて処理結果と詳細なエラーリストが画面上に返ること。
- 管理者がユーザー管理画面から各従業員のNFCカードを管理するダイアログを呼び出せる。
- 管理者が管理者ログイン状態で従業員のNFCカード情報（カード名、IDm）の一覧を参照・名前変更・削除できる。
- 管理者がユーザー管理画面でWebUSBを用いて読み取ったカード情報を従業員に対して登録できる。
- 出勤打刻から5分以内に退勤打刻を行った際、出退勤記録および監査ログが自動的に削除（取り消し）され、画面上に取り消された旨が示されること（クールダウン制限は適用したままとする）。
- 出勤打刻から5分以内に退勤打刻を行ったことになる（勤務時間が5分以下となる）修正申請は、フロントエンドおよびバックエンドでバリデーションされ、受け付けないこと。また、一般従業員による打刻修正申請は1分単位での指定が可能で、理由の入力が必須（空欄やスペースのみの入力はエラーとする）であること。
- 打刻修正申請が却下された際、申請者宛てにお知らせが却下コメント付きで自動作成されること。
- ログイン後のナビゲーションバーにお知らせベルと未読数バッジが表示され、クリックしてお知らせの閲覧や既読化ができること。
- 未ログインの打刻時、未読のお知らせがあるユーザーに対して打刻結果画面でお知らせの存在を示すアラートメッセージが表示されること。
- 作成から180日以上経過した古いお知らせが毎日 04:05 に自動クリーンアップ（削除）されること。
- ユーザーが完全削除された際、そのユーザー宛てのお知らせがすべて自動で削除（カスケード削除）されること。
- 管理者が、締め処理（ロック）されていない期間に従業員の勤怠記録を手動で追加・削除できること。
- すでに勤怠レコードが存在する日であっても、追加で別の時間帯のレコードを追加できること。
- 時間帯が重複する勤怠を登録しようとした際の警告メッセージ内の時間表記が、日本標準時（JSTローカル時刻）で表示されること。
- 複数打刻ペアが存在する場合、詳細カレンダーでそれぞれの勤怠エントリーに対応する打刻ソースが個別の行に分けて（エントリー毎に）表示されること。
- 管理者がデータベース全体のフルバックアップファイルを安全にダウンロード（エクスポート）できること。
- 管理者がバックアップファイルをアップロード（復元）し、整合性チェックとテーブル構成チェックを経てオンラインで安全にデータを上書き復元できること（復元実行前にデータが全て上書きされる旨の確認メッセージが表示されること）。
- 管理者がシステム設定画面から、シフトカレンダーの即時同期（「直ちに同期する」）を実行し、その実行結果（件数またはエラー詳細）が画面上で即時確認できること。
- API と DB が契約文書と整合する。
- **打刻表示時間と連続打刻割り込み**:
  - 管理者がシステム設定画面から打刻結果表示時間（`punch_result_display_seconds`）を 1〜300秒 の間で設定し、変更を保存できること。
  - 未ログイン時の打刻画面において、打刻完了またはエラーのメッセージが設定された秒数の経過後に自動的に消えること。
  - メッセージ表示中（タイムアウト待機中）であっても、次のカード打刻や代替打刻を受け付け（割り込み可能）、打刻開始時に既存のメッセージを自動クリアすること。
  - クールダウン中に発生した連続打刻エラーのメッセージについては、表示が即座にクリアされず、既存の成功結果を残したまま追加表示され、残りクールダウン秒数の経過後にその警告メッセージのみ自動で消去されること。
  - 5分以内の打刻取り消し（物理削除）の直後に再度打刻しようとした際にも、キャッシュされた打刻時刻によってクールダウン制限（`punch_cooldown_seconds`）が正しく適用され、不適切な出勤打刻の間違いを防止できること（10秒以内であればエラーにならず無視されること）。
- **打刻時の効果音再生**:
  - 打刻完了（出勤、退勤、打刻取消）および打刻エラー発生のタイミングで、対応する効果音（Web Audio APIによる動的合成音）がスピーカーから再生されること。
