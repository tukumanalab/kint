# 勤怠打刻判定規則（仕様書）

本ドキュメントでは、NFCカード（FeliCa IDm）を用いた打刻時の「出勤」「退勤」の自動判定ルール、および例外的な状況（シフト外の打刻、退勤忘れ）における処理方針を定義します。

## 1. 基本的な打刻判定（自動判定）

ユーザーがカードをかざした際、その時点でのユーザーの**出勤状態**に基づいて、システムが自動的に「出勤」または「退勤」を判定します。

*   **状態が「未出勤」の場合 ＝【出勤】として記録**
    *   その日のうちに完了した出退勤記録がすでにあっても、現在「未退勤の記録」が存在しなければ、新たな【出勤】として扱います。
    *   これにより、休憩等で一時的に退勤し、再度出勤するといった**1日に複数回の出勤/退勤**が可能となります。
*   **状態が「出勤中（未退勤）」の場合 ＝【退勤】として記録**
    *   直近の出勤記録に対して、退勤時間を打刻時間で更新（記録）します。

*   **【重要】連続打刻の防止（クールダウン）**
    *   誤ってNFCカードを長めにかざしてしまったり、連続でタッチしてしまったことによる「意図しない短時間での出勤/退勤の繰り返し」を防ぐため、**直近の打刻から一定時間（例：1分〜5分間）以内の新たな打刻は制限**します。
    *   ただし、誤タッチなどの即時の連続打刻に対応するため、**直近の打刻から10秒以内の連続打刻については、エラーを出さずに無視（正常系レスポンスを返すが、DBの更新は行わない）**します。
    *   10秒を超え、設定されたクールダウン時間以内の打刻の場合は、エラー（HTTP 409）として拒否し、フロントエンドには「直前に打刻したばかりです」といったメッセージを表示させます。
    *   クールダウン時間は環境変数 `PUNCH_COOLDOWN_SECONDS`（または管理画面のシステム設定）で調整します。

## 2. シフト外での出勤打刻（現場での確認）

【出勤】と判定された場合、システムはiCalのシフト情報と照合し、現在時刻がシフト予定の範囲内（※開始前N分〜終了時刻まで等）であるかを確認します。

*   シフト開始前許容（開始前N分）の値は環境変数 `SHIFT_CHECKIN_EARLY_MINUTES` で調整します。
*   iCal 同期元 URL は環境変数 `SHIFT_ICAL_URL` で管理します。

*   **シフト予定がある場合**
    *   そのまま通常の【出勤】として処理を完了します。
*   **シフト予定がない場合**
    *   バックエンド（API）は即座に記録を確定させず、フロントエンドに対して「確認要求（要確認ステータス）」を返します。
    *   フロントエンド（画面）はユーザーに対して、「**現在シフトが入っていませんが、出勤の打刻をしますか？**」といった確認ダイアログを表示します。
    *   ユーザーが画面上で「はい（確定）」を選択した場合、フロントエンドは「強制打刻フラグ（`confirm=true`等）」を付与して再度APIを呼び出し、【出勤】として記録を確定させます。

## 3. 退勤忘れ時の自動補完（確定仕様）

出勤の打刻をした後、退勤の打刻がないまま1日の営業日（翌朝午前4:00）が終了した場合、システムは毎日午前4:00に実行される夜間バッチ処理により、以下の基準で退勤時間を処理します。

*   **シフト予定がある場合の出勤**
    *   当該シフトの**「終了予定時刻」**を退勤時間として自動的に記録（補完）します。
    *   自動補完されたことを識別できるよう、`attendances` レコードの `is_auto_completed` フラグを `True` に設定し、`auto_completed_at` に補完実行日時を記録します。
    *   監査ログ（`AttendanceChangeLog`）を `actor_user_id='system'`（システム自動実行ユーザー）名義で自動登録します。
*   **シフト予定がない場合の出勤（シフト外で確認を経て出勤した場合）**
    *   終了時刻の推定が困難であるため、安全性の観点から**自動補完は行いません**。
    *   退勤時間は `NULL` のままとし、ステータスは「打刻不整合（`incomplete`）」のまま維持されます。後日、管理者や本人が手動で修正申請・修正を行う必要があります。

## 4. 処理フローの概要（バックエンド/フロントエンド）

1.  **NFCかざす** -> フロントエンドからAPIへ `POST /api/attendance/stamp` (IDm送信)
2.  **API判定**
    *   (事前チェック) 直近の打刻時刻からN分以内であれば -> **連続打刻エラー** -> エラーレスポンス
    *   未退勤の記録あり -> **退勤処理** -> 完了レスポンス
    *   未退勤の記録なし -> 出勤処理へ
3.  **出勤処理時のシフト確認**
    *   シフトあり -> **出勤処理** -> 完了レスポンス
    *   シフトなし -> `status: requires_confirmation` をレスポンス
    *   シフト外確認メッセージには、`SHIFT_ICAL_URL` の設定状態（設定あり/未設定）を含める
4.  **フロントエンドでの確認処理**（シフトなしの場合）
    *   画面にダイアログ表示「シフトがありません。打刻しますか？」
    *   ユーザー確認 -> `POST /api/attendance/stamp` (IDm + `confirm=true`) -> **出勤処理** -> 完了レスポンス
5.  **夜間バッチ処理**（毎日午前4:00実行）
    *   前日以前の「未退勤（出勤あり、退勤なし）」の記録を検索。
    *   その日にシフト予定が登録されている場合、シフト終了予定時刻を `check_out` にセットし、自動補完フラグ（`is_auto_completed=True`）を設定する。また、`system` ユーザー名義で監査ログを記録する。
    *   シフトがない場合は補完を行わずそのままスキップ（`incomplete` のまま）する。

## 5. 勤務時間の出勤/退勤時刻の丸め判定・算出

給与計算や労働時間算出の基盤となる「勤務出勤」および「勤務退勤」時刻は、iCal から同期されたシフト情報（開始時刻・終了時刻）に基づいて、実際の打刻時間（1分単位）から以下のルールで自動計算して算出・表示されます。

*   **【共通】秒の切り捨て（分単位への正規化）**: 丸め計算を行う前に、打刻時刻の**秒は常に切り捨て**て分単位に正規化します。そのうえで以下の5分単位の丸めを適用します。
    *   *例: 13:00:59 に打刻 → 秒を切り捨てて 13:00 として扱う → 勤務出勤は `13:00`*

### 5-1. 出勤時刻（勤務出勤）の丸めルール
*   **シフト開始前に打刻した場合**: シフトの開始時刻を出勤時刻として扱います。（切り下げ）
    *   *例: シフト 09:00 のときに 08:50 に打刻 → 勤務出勤は `09:00`*
*   **シフト開始後に打刻した場合**: 5分単位で切り捨てて（切り下げて）出勤時刻とします。
    *   *例: シフト 13:00 のときに 13:02 に打刻 → 勤務出勤は `13:00`*
    *   *例: シフト 13:00 のときに 13:05 に打刻 → 勤務出勤は `13:05`*
    *   *例: シフト 13:00 のときに 13:00:59 に打刻 → 秒を切り捨てて 13:00 として扱う → 勤務出勤は `13:00`*

### 5-2. 退勤時刻（勤務退勤）の丸めルール
*   **シフト終了前に打刻した場合**: 5分単位で切り上げて退勤時刻とします。
    *   *例: シフト 18:00 のときに 17:59 に打刻 → 勤務退勤は `18:00`*
*   **シフト終了後に打刻した場合**: シフトの終了時刻を退勤時刻として扱います。
    *   *例: シフト 18:00 のときに 18:04 に打刻 → 勤務退勤は `18:00`*

### 5-3. シフトがない日の丸めルール
*   **シフトが登録されていない日（公休・シフト外の強制打刻等）**:
    *   **出勤打刻 (check_in)**: 5分単位で切り捨てて勤務出勤時刻とします。
        *   *例: 09:02 に打刻 → 勤務出勤は `09:00`*
        *   *例: 09:05 に打刻 → 勤務出勤は `09:05`*
    *   **退勤打刻 (check_out)**: 5分単位で切り上げて勤務退勤時刻とします。
        *   *例: 18:04 に打刻 → 勤務退勤は `18:05`*
        *   *例: 18:05 に打刻 → 勤務退勤は `18:05`*

### 5-4. 労働時間・時間外時間の算出基準
*   システム内で集計される「労働時間（`working_hours`）」「時間外時間（`overtime_hours`）」、および CSV 出力される「出勤」「退勤」「勤務時間」（日次詳細CSV）は、すべて上記の丸め後の**「勤務出勤」「勤務退勤」時刻を基準として計算**されます。
*   これらは日別勤怠詳細カレンダー上で「打刻出勤/退勤」列とは独立した「勤務出勤/退勤」列として可視化されます。

## 6. 打刻修正申請と管理者による勤務時間の直接編集

打刻の正確性の維持と、管理者による柔軟な調整を両立するため、システムでは「実際の打刻時間」と「計算で使われる勤務時間」を切り分け、以下の編集・申請フローを提供しています。

### 6-1. 従業員による「打刻時間修正申請」
*   **対象データ**: 実際の打刻時間 (`check_in`, `check_out`)
*   **動作**:
    *   従業員は自分の「打刻時間」の修正申請 (`AttendanceCorrectionRequest`) を作成します。
    *   管理者がこの申請を「承認」すると、打刻時間が更新され、その打刻時間に基づいて「出勤/退勤時刻の丸め判定・算出（つくまなルール）」が**自動で再実行（再計算）**されます。
    *   この場合、手動編集フラグ (`is_manual_work_time`) は `False` のまま維持され、自動丸めの仕組みが継続して適用されます。

### 6-2. 管理者による「勤務時間の直接編集（追加・変更）」
*   **対象データ**: 勤務出勤/退勤時間 (`work_start`, `work_end`) 
    *   ※管理者は「実際の打刻時間」を直接書き換えることはできません。
*   **入力の制限**:
    *   手動追加・直接修正時の時刻入力は、「時」と「分」に分かれた選択式ドロップダウンメニューで行います。
    *   「分」の選択肢は **5分刻み**（`00`, `05`, `10`, ... `55`分）に制限されており、それ以外の分数は選択・保存できません。
    *   修正理由および追加理由の入力は**任意（オプショナル）**です（未入力で保存可能です）。
*   **動作**:
    *   管理者は日次詳細画面から、勤務出勤/退勤時間を直接指定して更新、または手動で追加することができます。
    *   管理者が直接編集を行うと、そのレコードは手動編集扱い (`is_manual_work_time = True`) となり、打刻時間からの自動丸め計算（つくまなルール）は行われず、**管理者が直接設定した勤務時間が最優先で保持**されます。
*   **自動計算への差し戻し（手動設定のクリア）**:
    *   管理者が手動設定した勤務時間をクリアし、「打刻からの自動計算（丸めルール）に戻したい」場合は、画面上の「自動計算に戻す」オプションを選択して保存します。
    *   これにより `is_manual_work_time` が `False` に戻り、実際の打刻時間を元に再度自動丸め計算が適用されます。

### 6-3. 管理者による「勤務時間の直接削除」
*   **打刻データが存在する日の削除**:
    *   その日に打刻データ (`check_in`, `check_out` のいずれかが登録されている状態) が存在する場合、勤怠レコード自体の物理削除は行いません。
    *   代わりに、`work_start` と `work_end` を `NULL` (勤務時間 0h) にクリアし、`is_manual_work_time = True` に設定します。これにより、実際の打刻記録を残しつつ、勤務時間は存在しない（削除された）状態として扱われます。
*   **打刻データが存在しない日の削除**:
    *   シフト予定しかなく打刻データが存在しない日に手動追加された勤務時間を削除する場合などは、勤怠レコード自体をデータベースから物理的に削除します。

### 6-4. 監査ログ（変更履歴）の記録
*   管理者が直接勤務時間を変更、削除、または自動計算に差し戻した場合、変更履歴 (`AttendanceChangeLog`) に変更前の勤務時間 (`before_work_start`/`before_work_end`) と変更後の勤務時間 (`after_work_start`/`after_work_end`) を詳細に記録します。
*   これにより、給与計算などの元データとなる勤務時間が、「いつ」「誰が」「どういう理由で」変更したかを完全に監査・追跡できるようになっています。
